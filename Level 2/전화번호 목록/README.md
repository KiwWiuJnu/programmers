# 전화번호 목록 - [해시 / 정렬]

## 1. 문제 개요 및 제한 조건
* **난이도**: 프로그래머스 Level 2
* **핵심 목표**: 전화번호부에 적힌 전화번호 중, 한 번호가 다른 번호의 접두어인 경우가 있는지 확인하기.
* **데이터 범위 및 시간 제한**:
  * $N$ (전화번호 개수): 최대 `1,000,000` (100만 개)
  * 문자열 최대 길이 $L$: `20`
  * **허용 시간 복잡도 유추**: $N$이 최대 100만 개이므로, $O(N^2)$ 알고리즘은 무조건 시간 초과(TLE)가 발생함. 최소 $O(N \log N)$ 또는 $O(N)$ 안으로 끊어야 함.

## 2. 나의 접근 단계 및 발전 흐름 (My Approaches)

### 1단계: 첫 번째 직관 & 설계 (First Attempt)
* **사용하려 했던 자료구조**: `std::map<string, int>`
* **설계 로직**: 
  1. 데이터를 자동으로 정렬시켜 주는 `std::map`에 데이터를 다 옮겨 담는다.
  2. 사전순 정렬의 특성상 "뒤에 있는 값은 절대 앞에 있는 값의 접두어가 될 수 없다"는 성질을 유치해 냄.
  3. 이중 루프를 통해 앞의 문자열(`i`)이 뒤의 문자열(`j`)의 접두어인지 `B.find(A) == 0`으로 탐색함.
* **결과**: 정확성은 성공했으나 효율성 테스트에서 **시간 초과(TLE)** 발생.
* **한계 및 오답 요인**: 
  * `map`은 정렬을 보장하지만 노드 삽입 비용이 크고, 이중 루프를 돌아 $O(N^2)$의 시간 복잡도가 발생하여 100만 개의 데이터를 해결하지 못함.

### 2단계: 최적화 단계 (Optimization)
* **수정된 논리**: 
  * 굳이 `map`을 새로 선언해서 복사하기보다, 기존 `vector`를 `std::sort()`로 정렬하면 사전식 순서 정렬이 완성됨.
  * 사전식 순서가 완료되었다면, **"어떤 문자열이 누군가의 접두어라면 반드시 바로 다음 칸(인접한 `i`와 `i+1` 위치)에 존재한다"**는 사실을 깨달음.
* **개선점**: 내 바로 뒷 칸의 데이터가 내 접두어가 아니라면 그 뒤의 데이터들도 절대 내 접두어가 될 수 없으므로, 이중 루프를 완전히 없애고 단일 루프를 통한 인접 비교(`phone_book[i]`와 `phone_book[i+1]`)로 변경함.

## ⏱️ 3. 성능 및 시간/공간 복잡도 분석
* **이중 루프 풀이**: $O(N^2 \cdot L)$ ➡️ 약 1조 번의 연산으로 **효율성 실패**
* **인접 정렬 풀이 (최종 통과)**: 
  * 정렬 단계: $O(N \log N \cdot L)$
  * 단일 루프 탐색 단계: $O(N \cdot L)$
  * 전체 시간 복잡도: **$O(N \log N \cdot L)$** ➡️ $100\text{만} \times 20 \times 20 \approx 4\text{억}$ 번 이하로 **효율성 가뿐히 통과**
* **정렬 vs 해시 성능 비교**:
  * **해시 풀이** ($O(N \cdot L)$): 모든 조각을 `unordered_map`에 넣어두고 접두어 조각을 잘라가며 탐색하는 방법. 정렬이 필요 없으나 해시 테이블 적재를 위한 추가 메모리가 필요함.
  * **정렬 풀이** ($O(N \log N \cdot L)$): 추가 메모리 소모가 극히 적어 메모리 제한이 엄격할 때 유용함.

## 4. 내가 범한 실수 & 예외 케이스 (Mistakes & Edge Cases)
* **find() 반환값 오해**: `find()`는 일치 여부를 `true/false`로 주는 것이 아니라, 일치하는 시작 인덱스를 반환하므로 접두어인지 검증하기 위해서는 **`== 0`**과 비교해야 함.

## 5. 핵심 디테일 & 새로 배운 개념 (Key Takeaways)

### 1) C++ `unordered_map`의 평균 $O(1)$ 탐색 원리
* `unordered_map`은 데이터를 찾을 때 방을 처음부터 뒤지지 않는다.
* 탐색 대상 문자열을 고유한 정수값으로 바꾸는 **해시 연산**을 수행하고, 배열의 크기로 나눈 나머지 연산(`%`) 결과를 인덱스 삼아 **해당 주소로 바로 순간이동**하기 때문에 $O(1)$의 기적적인 속도가 나온다.

### 2) `m.end()`와 `m.find()`의 실제 데이터 구조
* `m.end()`는 실제 데이터가 아니라, 컨테이너 영역의 **가장 마지막 데이터 바로 뒤의 가상 경계 주소(Iterator)**를 뜻함.
* `m.find(key)`가 키를 찾지 못하면 끝까지 헤매다가 결국 가상의 한계선인 `m.end()`에 도착해 버림. 따라서 `m.find(key) != m.end()` 구문이 **"끝에 도착하기 전에 찾았음(데이터가 존재함)"**이라는 의미를 갖게 됨.

### 3) 해시 충돌(Hash Collision)과 Chaining
* 해시 결과가 같은 방을 가리키는 충돌이 발생할 경우, C++ STL은 해당 버킷 내부를 **단방향 연결 리스트(Singly Linked List)**로 길게 줄 세우는 **Separate Chaining** 구조를 택함.
* 따라서 `find`는 단순 인덱스가 아니라 충돌 리스트 속에서 실제 원하는 Key를 찾아 해당 노드의 실제 메모리 주소(Iterator)를 반환하는 과정임.
