# 📝 [Programmers / Level 2] 전화번호 목록

> **분류**: 해시(Hash), 정렬(Sort)  
> **문제 링크**: [전화번호 목록](https://school.programmers.co.kr/learn/courses/30/lessons/42577)

## 1. 문제 파악 및 제한 조건
* **핵심 목표**: 전화번호부에 적힌 전화번호 중, 한 번호가 다른 번호의 접두어인 경우가 있는지 확인하기.
* **데이터 범위 및 제약 사항**:
  * $N$ (전화번호 개수): 최대 `1,000,000` (100만 개)
  * 문자열 최대 길이 $L$: `20`
* **시간 복잡도 유추**: $N$이 최대 100만 개이므로, $O(N^2)$ 알고리즘은 무조건 시간 초과(TLE)가 발생함. 따라서 최소 $O(N \log N)$ 또는 $O(N)$ 내에 해결해야 함.

## 2. 논리 전개 및 풀이 과정 (Approaches)

### 🚨 1차 시도 (접근 및 한계)
* **사용 자료구조**: `std::map<string, int>`
* **설계 로직**: 
  1. 데이터를 자동으로 정렬시켜 주는 `std::map`에 모두 삽입.
  2. "사전순 정렬 시, 뒤에 있는 값은 절대 앞에 있는 값의 접두어가 될 수 없다"는 성질 활용.
  3. 이중 루프를 통해 앞 문자열(`i`)이 뒤 문자열(`j`)의 접두어인지 `find() == 0`으로 탐색.
* **한계점 (면접 대비 포인트)**: 
  * 정확성은 통과했으나 **효율성 테스트 실패 (시간 초과)** 발생. `map`의 노드 삽입 비용이 크고, 무엇보다 이중 루프를 돌아 $O(N^2)$의 시간 복잡도가 발생하여 100만 개의 데이터를 감당하지 못함.

### ✅ 2차 시도 (최적화 및 최종 통과)
* **개선된 논리**: 
  * `map` 대신 기존 `vector`를 `std::sort()`로 정렬하여 사전식 순서를 확보함.
  * 사전식 정렬이 완료되었다면, **"어떤 문자열이 누군가의 접두어라면 반드시 인접한 위치(`i`와 `i+1`)에 존재한다"**는 점을 파악함.
  * 즉, 내 바로 뒷 칸의 데이터가 내 접두어가 아니라면 그 뒤의 데이터들도 절대 접두어가 될 수 없음.
* **결과**: 이중 루프를 완전히 없애고 단일 루프로 인접 비교만 수행하여 시간 초과 문제 해결.

## ⏱️ 3. 복잡도 분석 및 성능 비교
* **1차 시도 (이중 루프)**: $O(N^2 \cdot L)$ ➡️ 약 1조 번 연산으로 효율성 실패.
* **2차 시도 (인접 정렬)**: 
  * 정렬 단계: $O(N \log N \cdot L)$
  * 탐색 단계: $O(N \cdot L)$
  * **전체 시간 복잡도**: **$O(N \log N \cdot L)$** ➡️ $100\text{만} \times 20 \times 20 \approx 4\text{억}$ 번 이하로 통과.
* **💡 정렬 vs 해시 성능 비교 (최적화 면접 질문 대비)**:
  * **해시 풀이 ($O(N \cdot L)$)**: 모든 조각을 `unordered_map`에 넣고 잘라가며 탐색. 정렬은 필요 없으나 해시 테이블 적재를 위한 추가 메모리가 크게 발생함.
  * **정렬 풀이 ($O(N \log N \cdot L)$)**: 해시 풀이 대비 추가 메모리 소모가 극히 적어, 메모리 제한이 엄격한 환경에서 훨씬 유리한 최적화 방식임.

## ⚠️ 4. 엣지 케이스 및 실수 (Mistakes & Edge Cases)
* **`find()` 반환값 오해**: `find()`는 일치 여부(Boolean)가 아니라 **일치하는 시작 인덱스**를 반환함. 따라서 접두어인지 정확히 검증하려면 반드시 `== 0` 조건으로 엣지 케이스를 확인해야 함.

## 🧠 5. 핵심 디테일 및 딥다이브 (Key Takeaways)

### 1) C++ `unordered_map`의 평균 $O(1)$ 탐색 원리
* 데이터를 처음부터 순차 탐색하지 않음.
* 탐색 대상 문자열을 정수값으로 바꾸는 **해시 연산** 수행 후, 배열 크기로 나눈 나머지(`%`)를 인덱스 삼아 **해당 주소로 즉시 접근**하므로 기적적인 $O(1)$ 속도 달성.

### 2) 해시 충돌(Hash Collision)과 Chaining
* 서로 다른 키의 해시 결과가 같은 버킷을 가리키는 충돌 발생 시, C++ STL은 내부를 **단방향 연결 리스트(Singly Linked List)**로 연결하는 **Separate Chaining** 기법을 사용함.
* 따라서 탐색 시 충돌 리스트 내부에서 실제 Key를 찾아 메모리 주소(Iterator)를 반환함.

### 3) `m.end()`와 `m.find()`의 의미
* `m.end()`: 실제 데이터가 아닌, 컨테이너 영역의 **마지막 데이터 바로 뒤 가상 경계 주소(Iterator)**.
* `m.find(key) != m.end()`: 끝(가상 한계선)에 도착하기 전에 원하는 데이터를 찾았음, 즉 **"데이터가 존재함"**을 의미함.
