# 📝 [Programmers / Level 2] 의상

> **분류**: 해시(Hash), 수학(Math)  
> **문제 링크**: https://school.programmers.co.kr/learn/courses/30/lessons/42578

## 1. 문제 파악 및 제한 조건
* **핵심 목표**: 스파이가 가진 의상들이 주어졌을 때, 종류별로 조합하여 입을 수 있는 서로 다른 옷 조합의 수를 구하기. (단, 최소 한 개의 의상은 무조건 입어야 함)
* **데이터 범위 및 제약 사항**:
  * 의상의 수 $N$: 최대 30개
  * 같은 이름을 가진 의상은 존재하지 않음.

---

## 2. 논리 전개 및 풀이 과정 (Approaches)

### 🚨 1차 설계 (접근 및 한계)
* **초기 아이디어**: DFS(백트래킹)를 이용해 옷을 1개 입을 때, 2개 입을 때... 모든 조합(Combination)의 수를 구해서 더하는 방식을 구상함.
* **한계점**: 
  * 만약 의상의 개수($N$)가 많아질 경우, 멱집합을 구하는 것과 같아 시간 복잡도가 $O(2^N)$으로 폭증함. 
  * 이 방식은 불필요한 재귀 호출로 인해 치명적인 **시간 초과(TLE)** 및 메모리 초과를 유발할 수 있는 설계적 한계가 있음.

### ✅ 2차 시도 (수학적 최적화 및 최종 통과)
* **개선된 논리**: 
  * 데이터가 정렬될 필요가 없으므로 평균 탐색/삽입 속도가 $O(1)$인 `unordered_map<string, int>`을 사용하여 '옷의 종류'를 Key로, '개수'를 Value로 카운팅함.
  * 조합을 일일이 구하는 대신, **"해당 종류의 옷을 아예 입지 않는 경우(1가지)"**를 각 의상 개수에 더해주어(`pair.second + 1`) 곱의 법칙을 적용함.
* **결과**: 재귀 없이 1차원 `for`문만으로 모든 경우의 수를 한 번에 구하여 압도적으로 빠른 속도로 문제 해결.

## ⏱️ 3. 복잡도 분석 및 성능 비교
* **전체 시간 복잡도**: **$O(N)$**
  * 해시 맵에 데이터를 넣는 과정: $O(N)$
  * 해시 맵을 순회하며 경우의 수를 계산하는 과정: $O(K)$ ($K$는 옷의 종류, $K \le N$)
  * 입력 데이터 크기만큼만 순회하면 되므로 효율성 극대화.
* **공간 복잡도**: **$O(K)$**
  * 옷의 종류 개수만큼 해시 맵에 저장됨.

## ⚠️ 4. 엣지 케이스 및 한계점 (Mistakes & Edge Cases)
* **최소 1개의 의상 착용 (Edge Case)**: 
  * 수학적 조합으로 모두 곱해버리면 **"모든 종류의 옷을 하나도 입지 않은 상태(1가지)"**가 포함됨.
  * 스파이는 하루에 최소 한 개의 의상은 입는다는 조건이 있으므로, 최종 결과에서 반드시 `answer--`를 통해 이 예외 케이스를 배제해야 함.
* **자료형 오버플로우 한계점**:
  * 현재는 의상의 수가 최대 30개라 `int` 자료형(`최대 약 21억`)으로 충분하지만, 만약 옷의 종류와 개수가 수백~수천 개로 늘어난다면 누적 곱셈 과정에서 `int`의 범위를 초과할 수 있음. 
  * **개선 방안**: 제한사항이 큰 문제라면 `answer` 변수를 `long long` 타입으로 선언하여 오버플로우를 방지하는 것이 안전함.
* **해시 충돌 (Hash Collision) 한계점**:
  * `unordered_map`은 평균적으로 $O(1)$이지만, 최악의 해시 충돌 발생 시 $O(N)$까지 느려질 수 있음. 악의적인 테스트 케이스가 있는 환경에서는 삽입/탐색이 항상 $O(\log N)$을 보장하는 `std::map`을 고려할 수도 있음.

## 💡 5. 코드 리뷰 및 리팩토링 (Refactoring)
* **C++ 모던 문법(Range-based for) 활용**: 인덱스 기반의 `for`문 대신 범위 기반 `for`문을 사용하여 가독성을 높일 수 있음.
* **불변성 보장 (`const`)**: 값을 읽기만 하는 해시 맵 순회 시 `const auto&`를 사용하여 의도치 않은 값 변경을 방지하는 것이 좋음.
